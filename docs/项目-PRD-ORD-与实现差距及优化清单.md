# 项目：PRD / ORD / 流程图 与当前实现差距及优化清单

本文对照 **`产品需求文档 (PRD)`**、**`运营需求文档 (ORD)`**、**`从分析到落地的全链路执行清单`**、**`product_flow.png`**（用户动线）与仓库**当前代码**，列出差距、风险与建议优先级；并说明文档中 **「草案」** 一词的两种含义，避免与「未开发功能」混淆。

---

## 1. 「草案」在文档中的含义（不等价于「全部未实现」）

| 文档位置 | 含义 | 与代码关系 |
|----------|------|------------|
| **手册-D** §5「接口与数据模型（**研发草案**）」、§6「管理端 UI **草案**」 | 产品设计阶段的**规格草稿标题**，用于拆单 | **多数已落地**（`ai_insight_*` 表、管理端 `/admin/ai-seo-insights` 等） |
| **手册-D §11**、API 说明中的 **`page_seo_patch` 草案** | 工作流状态：**`ai_insight_seo_task.status = draft`**，须批准后再 apply | **已实现**；手工插入 API **`POST .../runs/{id}/seo-tasks`**；管理端 **`/admin/ai-seo-insights/runs/[id]`** 已提供 **「手工添加 page_seo 草案」** 表单 |
| **手册-C** 索引表「接口**草案**」 | 对 **手册-D（PROD-AI-SEO）** 的文档别名 | 能力以 **手册-D §8 / §11** 与代码为准 |

**结论**：不能把「文档里出现草案二字」一律理解为「还要从零开发」。**PRD 中大量能力从未标为草案**，但仍可能尚未在本仓库实现——以下第 2～4 节按**产品能力**逐项对照。

## 5. 建议的优化顺序（工程）

1. **P0**：安全与发布流程（**SEC-01、OPS-ENV**）— 见 **手册-A、手册-C**；后端推荐 **Python 3.11**（与 CI 一致）；**3.10+** 为手册约定下限；**3.9** 仅建议在装齐依赖（含 **`typing_extensions`**）且通过本机 `import app.main` 烟测后用于临时联调，勿用于生产。
2. **P1**：若 SEO 为核心增长：在已有 **`/s/*`、`/category/*`** 上排 **SSR/预渲染** 或与 `page_seo` 深度联动；非「从零做路由」。
3. **P1**：**真实 UGC 发评**：**POST** API + **`ReviewModal`** 对接 + **`ugc_status`/审核** 策略。
4. **P1～P2**：**开发者认领 + 审核**：表结构、邮件验证、管理端队列、（可选）开发者只读统计 API。
5. **P2**：**支付 + 自助订单 + 强曝光位**；与法务、计费模型绑定。
6. **持续**：`npm audit`、按需继续拆 **`lucide` 白名单**、PG 生产备份演练（**ENG-PG**）。
